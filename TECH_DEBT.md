## [TD-001] Мягкое удаление члена при отмене проведения документа владения

**Обнаружено:** 2026-05-21  
**Связано с:** `private.members`, `private.doc_ownership`

**Проблема:**  
При удалении или отмене проведения `doc_ownership` поле `members.source_doc_id` становится NULL (ON DELETE SET NULL), но сама запись члена остаётся активной (`is_active = true`). Человек формально числится членом организации, хотя владение аннулировано.

**Когда важно:**  
Когда заработает валидация назначения в ревкомиссию (audit_member должен быть активным членом) — этот кейс станет источником ошибок.

**Варианты решения:**  
- При unpost документа владения — устанавливать `members.is_active = false` и `members.left_at = doc_date`
- Либо проверять не `members.is_active`, а наличие активного doc_ownership для данного контрагента напрямую

## [TD-002] Валидация назначения должностных лиц

**Обнаружено:** 2026-05-21  
**Связано с:** `private.org_officers`, RPC `api.set_org_officer`

**Проблема:**  
В MVP (migration 023) RPC `set_org_officer` не проверяет права назначения. Правила:
- `audit_member`: должен быть активным членом организации (`private.members.is_active = true`)
- `chairman`, `treasurer`: ограничений нет

**Код ошибки при реализации:** `NOT_A_MEMBER: <contractor_id>`

**Зависит от:** TD-001 (сначала надо корректно отрабатывать мягкое удаление члена)
