# Дизайн: Право владения участком

**Дата:** 2026-05-12  
**Статус:** Согласован  
**Проект:** Controlling (система управления СТ)

---

## 1. Задача

Оформление права владения участком — ключевая операция для казначея СТ. Когда новый человек покупает участок или меняется владелец, казначей должен:

1. Найти или создать контрагента (физлицо или юрлицо)
2. Оформить документ «Право владения»
3. Провести документ — система автоматически обновляет реестр финансовых объектов и создаёт члена СТ (если физлицо)

---

## 2. Ключевые решения

### Реестр владения
`financial_object_registry` — периодический регистр (аналог РС Периодический в 1С). Хранит полную историю владения. Каждая строка: `(object_type, object_id, contractor_id, valid_from, valid_to)`. Текущий владелец — строка с `valid_to IS NULL`.

Таблица `plot_ownerships` — удаляется (дублировала реестр).

### Отмена документа
Отмены нет. Паттерн — append-only: если оформили не тому владельцу, создаётся новый документ на правильного. Предыдущая запись в реестре автоматически закрывается. Это правильно для СТ: владение меняется редко, история важна, сторно избыточно.

### Автосоздание члена СТ
Происходит автоматически при проведении документа, если:
- Контрагент — физлицо (`contractor_type = 'individual'`)
- У этого контрагента ещё нет активного члена СТ в данной организации

Номер члена СТ — порядковый, уникальный в рамках организации, назначается автоматически (`MAX(member_number) + 1`). Три уровня защиты от дублей: проверка типа → NOT EXISTS → UNIQUE-ограничение в БД.

### Контрагент vs Владелец
В интерфейсе используется слово «Владелец». Внутри — `contractors` с полем `contractor_type` (individual / legal_entity). Юрлицо не получает членство в СТ.

## 4. RPC-функции (PostgREST)

### `search_contractors(org_id, query)`
Полнотекстовый поиск по контрагентам организации. Возвращает: id, full_name, contractor_type, phone. Используется для живого поиска в диалоге.

### `create_contractor(org_id, full_name, contractor_type, phone, email, address)`
Создаёт нового контрагента. Возвращает его id.

### `create_ownership(org_id, contractor_id, object_type, object_id, doc_date, notes, created_by)`
Создаёт документ в статусе `draft`. Возвращает doc_id. Не меняет реестр — только создаёт документ.

### `post_ownership(doc_id)`
Проводит документ. Выполняется в транзакции:
1. Закрыть текущую запись реестра: `UPDATE financial_object_registry SET valid_to = doc_date - 1 WHERE object_type = ... AND object_id = ... AND valid_to IS NULL`
2. Создать новую запись: `INSERT INTO financial_object_registry (valid_from = doc_date, valid_to = NULL, contractor_id = ...)`
3. Если `contractor_type = 'individual'` И нет активного члена → `INSERT INTO members (member_number = MAX+1, source_doc_id = doc_id)`
4. `UPDATE doc_ownership SET status = 'posted'`

## 5. Фронтенд

### Точка входа
Страница «Участки» (`PlotsPage`):
- Кнопка «+ Оформить владение» в заголовке страницы (открывает диалог, участок выбирается внутри)
- Кнопка «Назначить владельца» на карточке каждого участка без текущего владельца (открывает тот же диалог, участок предвыбран)

### Компонент `OwnershipDialog`

Один модальный диалог с тремя блоками:

**Блок 1 — Участок**
- Дропдаун со списком участков без текущего владельца
- Если открыт с карточки участка — поле предзаполнено

**Блок 2 — Владелец**
- Строка поиска: ввод → живые подсказки через `search_contractors`
- Нашёл → выбрал → подтверждено
- Не нашёл → «Создать нового»: форма с полями: тип (физлицо / юрлицо), ФИО или название, телефон
- Член СТ создаётся автоматически — пользователь это не видит

**Блок 3 — Дата и заметки**
- Дата документа (по умолчанию сегодня)
- Заметки (необязательно)
- Кнопка «Провести документ»

**После проведения**
- Toast-уведомление: «Право владения оформлено»
- Диалог закрывается
- Список участков обновляется (владелец отображается на карточке)

## 6. Граничные случаи

| Ситуация | Поведение |
|----------|-----------|
| Участок уже имеет владельца | Предупреждение в диалоге: «Участок уже закреплён за [ФИО]. Продолжить — создать новый документ и переоформить.» |
| Контрагент — юрлицо | Член СТ не создаётся, всё остальное как обычно |
| У физлица уже есть членство | Второй член СТ не создаётся, `source_doc_id` не обновляется |
| Два казначея одновременно проводят на один участок | UNIQUE-ограничение в `financial_object_registry` (через функцию) или строчная блокировка в транзакции |
