# Ownership Flow Redesign

**Дата:** 2026-05-15  
**Статус:** Согласован  
**Приоритет:** Высокий

---

## Цель

Исправить работу с документами владения участками:
- Seed создаёт только справочные данные (без документов)
- Документы владения проводятся через RPC, не через прямую запись в БД
- Добавить возможность отменять документы (с каскадом и датой актуальности)
- Убрать избыточный реестр — документ владения сам является реестром

---

## Что меняется

### 1. Дата актуальности (по организации)

Добавляем поле `actuality_moment TIMESTAMPTZ` в таблицу организаций.

- Провели документ → `actuality_moment = MAX(текущая, posted_at документа)`
- Отменили документ → `actuality_moment = posted_at отменённого - 1 миллисекунда`

Показывает: «до этого момента всё проведено и актуально».

### 2. Документы — единый журнал

Каждый документ владения создаёт запись в общей таблице `documents` (тип = `'ownership'`).  
Это даёт единый журнал для каскадной отмены по дате.

Связь: `doc_ownership.document_id → documents.id`

### 3. Документ владения = реестр

Отдельный реестр (`financial_object_registry`) для владения **не нужен**.  
Текущий владелец участка = последний проведённый `doc_ownership` по этому участку.

Запрос текущего владельца:
```sql
SELECT contractor_id, shares
FROM private.doc_ownership
WHERE organization_id = :org_id
  AND object_id = :plot_id
  AND status = 'posted'
ORDER BY doc_date DESC, created_at DESC
LIMIT 1
```

`api.plot_summary` переписывается под этот запрос.

### 4. Доли владения

В `doc_ownership` добавляем поле `shares INTEGER DEFAULT 1`.

Расчёт доли = `shares / SUM(shares)` по всем записям одного документа.  
Пока всегда 1 владелец и 1 доля. Механизм готов для совладения.

Ключ уникальности записи: `(organization_id, object_id, contractor_id, doc_date)` — один владелец не может владеть одним участком дважды в одну дату.

### 5. Отмена документа (unpost_ownership)

Новый RPC `api.unpost_ownership(p_doc_id UUID)`:

1. Загружает документ, получает `organization_id`, `doc_date`, `posted_at`
2. Проверяет: `doc_date > дата закрытого периода` (иначе запрет)
3. **Каскад:** все записи в `documents` для этой организации, где `posted_at >= этого документа posted_at` → статус `'draft'`
4. **Синхронизация:** соответствующие `doc_ownership` записи → статус `'draft'`
5. **Дата актуальности:** `organizations.actuality_moment = posted_at - 1 мс`

Документы не удаляются — только статус. Пользователь проводит их заново.

### 6. Изоляция организаций

- PostgreSQL RLS уже изолирует данные по `organization_id` на уровне БД
- Все функции дополнительно фильтруют только по своей организации
- Никакая операция не может затронуть другую организацию

## Что делаем по шагам

### Миграция 011

1. `documents.doc_type` — добавить `'ownership'` в CHECK constraint
2. `doc_ownership` — добавить поля:
   - `document_id UUID REFERENCES private.documents(id)`
   - `shares INTEGER NOT NULL DEFAULT 1`
3. `organizations` — добавить `actuality_moment TIMESTAMPTZ`
4. Обновить `api.create_ownership` — также создаёт запись в `documents`
5. Обновить `api.post_ownership` — обновляет `documents.status`, `documents.posted_at`, `organizations.actuality_moment`
6. Новый RPC `api.unpost_ownership` — каскадная отмена с обновлением actuality_moment
7. Обновить `api.plot_summary` — запрос текущего владельца через `doc_ownership`, не через `financial_object_registry`

### Seed (исправление 007)

Убираем из seed:
- Прямую запись `owner_id` в таблицу участков
- Вставку в `plot_ownerships` (таблица уже удалена в 010)

Seed создаёт только справочники:
- 2 организации: СТ «Демо-А» и СТ «Демо-Б»
- Каждая: 5 участков, 5 физлиц, 2 юрлица
- Пользователи (без изменений)

Документы владения — пользователь создаёт вручную через RPC для тестирования.

### init.sh (обновление)

Добавить применение миграций 008, 009, 010, 011 после 007.

### Текущая база

Удалить все существующие организации (каскадом), запустить исправленный seed заново.

## Что НЕ входит в эту задачу

- Каскадное автопроведение (sequential reposting) — отдельная задача
- Совладение (несколько совладельцев с долями) — механизм заложен, реализация отдельно
- Корректировочные документы для закрытого периода — отдельная задача

---

## Изоляция данных (контракт)

Любая функция получает на вход `doc_id` → из документа берёт `organization_id` → все операции только по этой организации. Физически затронуть другую организацию невозможно: RLS запретит.
